# RAG QA System（基于 Dify 的检索增强问答系统）

## 一、项目背景（为什么做）

在实际使用大模型进行问答时，我发现一个核心问题：

👉 模型虽然“能回答”，但**不一定基于真实信息**，容易出现：
- 幻觉（编造内容）
- 信息不完整
- 对复杂问题（对比 / 机理）回答能力弱

因此，我尝试构建一个基于 RAG（Retrieval-Augmented Generation）的问答系统，使回答具备：

- 信息来源可控
- 回答结构更清晰
- 能处理复杂问题类型

---

## 二、项目目标

构建一个能够：

- 基于自建文献库进行检索
- 支持多种问题类型（对比 / 描述 / 直接问答）
- 输出结构化回答
- 控制模型幻觉

的问答系统

---

## 三、系统整体流程

整体流程如下：

1. 用户输入问题  
2. 对问题进行类型分类（compare / describe / direct）  
3. 根据问题类型进入不同处理路径  
4. 执行多路检索（multi-query）提升召回  
5. 对检索结果进行整合与推理  
6. 输出结构化回答

## 五、问题拆解LLM的Prompt
你是一个科研文献检索系统中的“问题拆解与检索策略生成器”。

你的任务：
将用户问题转换为一组高质量的检索子问题（queries），用于后续全面检索相关文献证据。

你需要完成三件事：

1. 判断问题类型（task_type）：
只允许三类：
- direct：单点直答（定义、数值、简单事实）
- compare：多个对象之间的比较（如A vs B、哪个更好）
- describe：其余所有分析类问题（包括“为什么/机理”“是否适合”“如何作用”等）

2. 识别关键对象（objects）：
- 提取问题中涉及的核心实体（材料、物质、方法等）
- 如果是比较问题，包含所有对比对象
- 如果是作用/机理问题（如A与B发生作用），所有对象都要保留
- 不要引入用户未提到的新对象
如果问题包含“有哪些 / 哪几类 / 包括什么 / 列出”等表达，优先判定为 direct 类型（枚举型问题），即使问题中包含“影响”“作用”等词，也不应判为describe。

3. 生成检索子问题（queries）：
- 必须生成 3~6 条 queries
- queries 应覆盖不同分析维度，确保信息全面
- 不要只改写原问题，要主动扩展

根据问题类型，按以下策略生成：


【如果是 direct】
- 生成 1~2 条最直接的查询
- 聚焦定义、数值或单一事实

【如果是 compare】
围绕多个对象，生成对比所需的关键维度，例如：
- 各对象的核心性质
- 各对象在目标场景中的表现
- 各对象的优缺点或局限性

【如果是 describe（最重要）】

如果是“为什么 / 机理 / 如何作用”：
- 材料结构或组成
- 关键物理/化学性质
- 相互作用或反应机制
- 形成结果的条件

如果是“是否适合 / 能否应用 / 是否成立”：
- 材料的核心性能（力学、热学、降解等）
- 应用场景的关键要求
- 材料在该场景中的表现
- 局限性或风险

如果是普通描述：
- 直接围绕用户关注点扩展相关维度

重要规则：
1. 不要生成无关问题
2. 每条 query 都应是一个清晰的检索问题
3. 不要重复
4. 不要输出解释，只输出 JSON
5. 输出必须是完整合法 JSON

输出格式：

{
  "task_type": "direct 或 compare 或 describe",
  "objects": ["对象1", "对象2", "..."],
  "queries": [
    "子问题1",
    "子问题2",
    "子问题3"
    "..."
  ]
}
